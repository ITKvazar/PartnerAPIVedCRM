Партнёрское API сервиса VedCRM для интеграции по ключам.

Назначение

API предназначено для внешних партнёров, которые интегрируются с платформой VedCRM для работы со справочником валют, контрактами и заявками на трансграничные платежи (международные переводы по экспортным/импортным/транзитным контрактам).

Через это API партнёр может:

  • получать справочник валют и их курсы;
  • получать список своих контрактов и детальную информацию по ним;
  • предварительно рассчитывать сумму к перечислению (deposit) и комиссию;
  • создавать заявки на платёж;
  • прикладывать к заявкам документы;
  • получать статус заявок (через периодический опрос — polling);
  • скачивать акт по заявке в формате PDF.

Авторизация

Авторизация выполняется по протоколу OAuth2 Client Credentials.

  1. Партнёр получает пару client_id / client_secret (выдаётся в рамках внутреннего процесса онбординга, вне данного API).
  2. Партнёр обращается к POST /auth/token с grant_type=client_credentials, передавая client_id и client_secret, и получает access_token (TTL ~4 часа) и refresh_token (TTL ~2 дня).
  3. Все последующие запросы выполняются с заголовком Authorization: Bearer <access_token>.
  4. По истечении срока действия access_token партнёр обновляет пару токенов через POST /auth/token/refresh, не повторяя полную аутентификацию.

Все методы, кроме POST /auth/token и POST /auth/token/refresh, требуют действующий bearer-токен.

Доступ к чужим ресурсам

Партнёр работает только со своими контрактами и заявками. При обращении к ресурсу, который не принадлежит партнёру, API намеренно возвращает 404 Not Found (а не 403), чтобы не раскрывать факт существования чужих объектов. Отдельного ответа 403 Forbidden в API нет.

Ограничение частоты запросов (rate limiting)

Часть методов (в частности polling-методы GET /applications и GET /applications/{application_id}) защищены ограничением частоты запросов. При превышении лимита возвращается 429 Too Many Requests. Рекомендуемый интервал опроса — не чаще одного запроса в 30–60 секунд на заявку.

Модель отслеживания статуса (polling)

Вебхуки не поддерживаются. Статус заявки отслеживается партнёром самостоятельно путём периодического опроса методов GET /applications и GET /applications/{application_id}. Рекомендуемый интервал опроса — не чаще одного запроса в 30–60 секунд на заявку.

Жизненный цикл заявки

  1. (Опционально) GET /applications/calculate-deposit — предварительный расчёт.
  2. POST /applications — создание заявки. Заявка создаётся в статусе on_inspect.
  3. POST /applications/{application_id}/documents — приложение документов к созданной заявке (порядок: сначала заявка, затем документы).
  4. GET /applications/{application_id} — опрос статуса.
  5. GET /applications/{application_id}/act — получение акта в PDF, когда заявка дошла до соответствующего статуса.

Партнёр не изменяет и не отменяет заявку — соответствующих методов в API нет.

Соглашения

  • Все денежные/decimal-поля передаются строкой (type: string) с ограничением формата через pattern, поскольку format: decimal не валидируется стандартными OpenAPI-валидаторами.
  • Для свойств, ссылающихся на enum-схемы через $ref, пример берётся из самой enum-схемы (example внутри схемы перечисления). Это считается достаточным для требования «у каждого поля есть example».

Auth

Аутентификация по OAuth2 Client Credentials. Получение и обновление пары токенов (access + refresh). Методы этого раздела не требуют bearer-токена.

Получить пару токенов (Client Credentials)

Выдаёт пару токенов (access_token + refresh_token) по учётным данным партнёра. Используется grant client_credentials. Тело передаётся в формате application/x-www-form-urlencoded согласно спецификации OAuth2.

access_token действует ~4 часа (expires_in = 14400), refresh_token — ~2 дня (refresh_expires_in = 172800). Все защищённые методы вызываются с заголовком Authorization: Bearer <access_token>.

Обновить пару токенов

Выдаёт новую пару токенов (access_token + refresh_token) по действующему refresh_token. Используется grant refresh_token. Полная повторная аутентификация по client_secret не требуется, но client_id передаётся для идентификации партнёра.

Если refresh_token истёк или недействителен — возвращается 401, и партнёр должен заново получить токены через POST /auth/token.

Предварительный расчёт суммы к перечислению и комиссии

Создать заявку

Создаёт заявку на платёж. Заявка создаётся в статусе on_inspect (на проверке). При успешном создании возвращается 200 OK (HTTP-код успеха не переопределён на стороне сервиса).

Особенности валидации:

  • currency_code сопоставляется с валютой в системе; курс берётся из справочника.
  • Должно выполняться exchange_rate == deposit / amount (курс из справочника обязан соответствовать переданным deposit и amount).
  • Для всех валют, кроме RUB, поле deposit обязано отличаться от amount. Для RUB это ограничение не применяется.
  • contract_number сопоставляется с контрактом партнёра.

После создания заявки документы прикладываются отдельным методом POST /applications/{application_id}/documents.

Приложить документ к заявке

Прикладывает документ к ранее созданной заявке. Передаётся как multipart/form-data. Порядок работы: сначала создаётся заявка (POST /applications), затем к ней прикладываются документы.

Authorizations:
bearerAuth
path Parameters
application_id
required
string <uuid>
Example: 770e8400-e29b-41d4-a716-446655440010

Идентификатор заявки.

Request Body schema: multipart/form-data
required
file
required
string <binary>

Бинарное содержимое файла документа (примеру не подлежит).

file_name
required
string

Имя файла.

type
required
string (DocumentTypeEnum)
Enum: "SWIFT" "ACT_REPORT" "PAYMENT_EXECUTION" "PAYMENT_ORDER" "VBK" "RELATIONSHIP_CONTRACT" "DEAL_CONTRACT" "IMPORT_APPLICATION" "EXPORT_APPLICATION" "INVOICE" "SPECIFICATION" "ATTACHMENT" "PACKING_LIST" "CUSTOMS_DECLARATION" "AIR_WAYBILL" "PERFORMANCE_BOND" "AGENCY_AGREEMENT" "COMPANY_CARD" "FOREIGN_TRADE_CONTRACT"

Тип документа (значения в верхнем регистре). Расшифровка значений:

  • SWIFT — SWIFT-сообщение/подтверждение перевода;
  • ACT_REPORT — акт (отчёт) по заявке;
  • PAYMENT_EXECUTION — подтверждение исполнения платежа;
  • PAYMENT_ORDER — платёжное поручение;
  • VBK — ведомость банковского контроля;
  • RELATIONSHIP_CONTRACT — рамочный договор об отношениях;
  • DEAL_CONTRACT — контракт сделки;
  • IMPORT_APPLICATION — заявление на импорт;
  • EXPORT_APPLICATION — заявление на экспорт;
  • INVOICE — счёт-фактура (инвойс);
  • SPECIFICATION — спецификация;
  • ATTACHMENT — произвольное вложение;
  • PACKING_LIST — упаковочный лист;
  • CUSTOMS_DECLARATION — таможенная декларация;
  • AIR_WAYBILL — авианакладная;
  • PERFORMANCE_BOND — гарантия исполнения обязательств;
  • AGENCY_AGREEMENT — агентский договор;
  • COMPANY_CARD — карточка компании;
  • FOREIGN_TRADE_CONTRACT — внешнеторговый контракт.
